How this simulation works

This playground never contacts a real HookGuard gateway. Every scenario above is a fixed, pre-written example — the accepted/rejected outcome and the reason shown are scripted in this page's JavaScript, not computed by running real HMAC/RSA verification in your browser. That's deliberate: this project's actual correctness proof is a differential test harness that runs real Go crypto against each provider's own official library, not a JavaScript reimplementation that would just be one more thing to keep in sync — and to trust.

Nothing on this page sends a network request. The "Simulated request" box shows realistic example headers and bodies in each provider's real format, but the signatures shown are illustrative strings, not values computed from an actual secret.

The rejection reasons here are real

The reason text next to a rejected scenario (signature mismatch, stale timestamp, cert host rejected, …) is the exact taxonomy the real gateway emits for every verdict it makes — see classifyReason in the root events.go. This page reuses that language rather than inventing demo-only wording, so what you read here matches what you'd see in the Console's Live Logs on a real deployment.

Want the real thing, not a simulation?

  • Read the actual verifier source — stripe.go, github.go, shopify.go, paypal.go — each is under 220 lines.
  • Run the differential harness yourself: go test -v -run Differential . — it checks each verifier's verdict against the provider's own official library (stripe-go, go-github), not just against itself.
  • Deploy it for real: the quick-start above, or the full production checklist.
